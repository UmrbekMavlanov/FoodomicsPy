# FoodomicsPy Developer Guide**Version:** 0.1.0---# 1. PhilosophyFoodomicsPy is an open-source Python package for foodomics, spectroscopy, chemometrics, metabolomics, lipidomics, and analytical chemistry.The primary goal is to make scientific data analysis:* Simple* Readable* Reproducible* Well documented* Publication readyThe package is intended for researchers who may not be professional software developers. Therefore, code readability is always more important than clever programming.Whenever there is a choice between a shorter but difficult implementation and a slightly longer but clearer implementation, choose the clearer one.---# 2. Package DesignThe package follows a module-based API.Users should interact with logical modules instead of hundreds of top-level functions.Example:```pythonimport foodomicspy as fpX = fp.normalization.minmax(X)X = fp.baseline.asls(X)model = fp.chemometrics.pca(X)fp.plotting.scores_plot(model)```Avoid exposing large numbers of functions directly under```pythonfp.function()```Instead use```pythonfp.module.function()```This keeps the API organized and scalable.---# 3. Project Structure```FoodomicsPy/src/    foodomicspy/        io.py        preprocessing.py        normalization.py        baseline.py        nmr.py        ftir.py        chemometrics.py        statistics.py        plotting.py        validation.py        utils.pytests/examples/docs/```Each module should represent one scientific topic.---# 4. Naming ConventionsPrefer short scientific names.Good:```pythonminmax()snv()msc()pqn()pca()pls()opls()anova()ttest()phase_correct()cut_region()```Avoid unnecessarily long names.Bad:```pythonnormalize_data_between_zero_and_one()perform_principal_component_analysis()perform_phase_correction_on_spectrum()```---# 5. Variable NamesUse consistent variable names throughout the package.| Variable | Meaning                                       || -------- | --------------------------------------------- || X        | Data matrix                                   || y        | Response vector                               || x        | X-axis (ppm, wavelength, wavenumber, time...) || axis     | Calculation axis                              || copy     | Whether input should be copied                || labels   | Sample labels                                 || groups   | Experimental groups                           |Spectral matrices should always follow```Rows    = SamplesColumns = Variables```Never change this convention.---# 6. Function DesignEvery public function should* perform one task only* have descriptive arguments* validate inputs* return results* never print results* include documentation* include an example* have at least one unit test---# 7. Docstring StyleUse NumPy-style docstrings.Example```pythondef minmax(X, axis=1):    """    Scale data to the range 0–1.    Parameters    ----------    X : array-like        Input data.    axis : int, default=1        Axis used for scaling.    Returns    -------    ndarray        Scaled data.    """```---# 8. Error HandlingAlways fail with meaningful messages.Good```pythonraise ValueError(    "X must be a 2D array with shape (samples, variables).")```Bad```IndexErrorTypeErrorRuntimeError```without explanation.---# 9. Return ValuesFunctions should return results instead of printing.Good```pythonscores = fp.chemometrics.pca(X)```Bad```pythonprint(scores)```---# 10. PlottingPlotting functions should return```pythonfig, ax```instead of calling```pythonplt.show()```internally.This allows users to continue editing publication figures.---# 11. Module Responsibilities## io.pyReading and writing files.Examples* read_csv()* read_excel()* read_hdf5()* save_excel()---## preprocessing.pyGeneral preprocessing.Examples* cut_region()* smooth()* derivative()* binning()---## normalization.pyNormalization and scaling.Examples* minmax()* max()* area()* vector()* snv()* pqn()* msc()---## baseline.pyBaseline correction.Examples* polynomial()* asls()* airpls()* arpls()* rubberband()---## nmr.pyNMR-specific processing.Examples* reference()* phase_correct()* integrate()* peak_pick()* align()* icoshift()---## ftir.pyFTIR-specific processing.Examples* read_opus()* atr_correction()* trans_peak()---## chemometrics.pyMultivariate analysis.Examples* pca()* pls()* opls()* mcr()* asca()* hca()---## statistics.pyStatistics.Examples* anova()* tukey()* ttest()* correlation()---## plotting.pyPublication-quality visualization.Examples* spectra_plot()* scores_plot()* loadings_plot()* heatmap()* scree_plot()---## validation.pyValidation metrics.Examples* rmse()* r2()* lod()* loq()* cross_validate()---## utils.pyInternal helper functions.Examples* check_array()* to_numpy()* validate_axis()---# 12. DependenciesPrefer* NumPy* pandas* SciPy* scikit-learn* matplotlibAvoid introducing new dependencies unless they provide significant value.---# 13. Backward CompatibilityOnce a function becomes public```pythonfp.normalization.minmax()```its behavior should remain stable across future releases whenever possible.Breaking changes should only occur in major releases.---# 14. TestingEvery important public function should have a unit test.Example```tests/test_normalization.pytest_baseline.pytest_nmr.py```Tests should verify* correct output* expected errors* edge cases---# 15. ExamplesEvery public function should include a small usage example.Users should understand a function without reading its implementation.---# 16. DocumentationEvery module should eventually contain* API documentation* Examples* References* Scientific background where appropriate---# 17. VersioningSemantic versioning will be used.```0.1.0Initial package skeleton0.2.0Core preprocessing0.3.0NMR tools0.4.0FTIR tools0.5.0Chemometrics1.0.0Stable public release```---# 18. Golden RuleBefore merging any new function, ask:* Is the name simple?* Is the implementation readable?* Is it documented?* Is it tested?* Does it match the API philosophy?* Will a scientist understand how to use it?If the answer to any of these questions is "No", improve the function before adding it to FoodomicsPy.
